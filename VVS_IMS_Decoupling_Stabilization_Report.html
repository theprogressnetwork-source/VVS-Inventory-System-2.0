01. Decoupling Mission Statement & "Ghost Inventory" Resolution

To establish a robust dual-tracking atomic inventory engine for VVS Computers, we have executed an exhaustive refactoring of the backend. In past iterations, the application services were heavily coupled with `VvsImsDbContext`. This monolithic dependency caused system locks, "ghost inventory" leakage, and direct pipeline exposures to the database engine.

The newly isolated system ensures that the **Application layer** communicates with raw database sets exclusively through the generic `IRepository` and `IUnitOfWork` interfaces. Data mutation pipelines are fully segregated and transactional.

By implementing this decoupling, database schema changes no longer cause cascade compile failures inside the core inventory synchronization logic. In addition, the **Api Controller layer** (`StockController.cs`) has been successfully refactored to remove all direct database context dependencies, forcing cleanly routed mapping flows.

This ensures that Best Buy and Amazon API operations and webhook callbacks remain highly responsive, completely type-safe, and free from concurrent database locks.

03. Decoupled C# Service Architecture — Transaction Integrity

C# / ASP.NET CORE 10.0

Below is the hardened implementation details applied to the C# application layer, using unit-of-work repositories and dynamic transaction boundaries to maintain ACID compliance without leaking concrete Entity Framework storage schemas.

C# / Decoupled Code 
/// <summary>
/// Process incoming scanning event and bind physical device IMEI
/// Uses Repository and Unit of Work decoupling for full ACID safety.
/// </summary>
[HttpPut("imei")]
public async Task<ActionResult<ApiResponse<object>>> UpdateImei([FromBody] StockUpdateRequest request, CancellationToken ct)
{
    _logger.LogInformation($"Binding physical IMEI {request.Imei} to order {request.OrderNo}");

    // 1. Begin atomic transaction using Unit of Work abstraction
    using var transaction = await _unitOfWork.BeginTransactionAsync(ct);
    try
    {
        // 2. Query placeholders utilizing Generic Repository abstractions
        var placeholderStock = await _stockRepo.Query
            .FirstOrDefaultAsync(s => s.OrderNo == request.OrderNo && s.Imei.StartsWith("P-") && !s.IsShipped, ct);

        if (placeholderStock == null)
        {
            var stock = await _stockRepo.GetByImeiAsync(request.Imei, ct);
            if (stock is null) return NotFound(ApiResponse<object>.Fail("Stock item not found by IMEI"));

            if (!string.IsNullOrWhiteSpace(request.NewImei)) stock.Imei = request.NewImei;
            _stockRepo.Update(stock);
            await _unitOfWork.SaveChangesAsync(ct);

            // Dynamic commit protects layer isolation
            await ((dynamic)transaction).CommitAsync(ct);
            return Ok(ApiResponse<object>.Ok(null, "IMEI updated directly"));
        }

        // 3. Assign physical device and update auxiliary repositories
        placeholderStock.Imei = request.Imei;
        _stockRepo.Update(placeholderStock);

        var outgoingRepo = _unitOfWork.Repository<Outgoing>();
        var outgoing = await outgoingRepo.Query
            .FirstOrDefaultAsync(o => o.OrderNo == request.OrderNo && (o.Imei == null || o.Imei.StartsWith("P-")), ct);
        if (outgoing != null)
        {
            outgoing.Imei = request.Imei;
            outgoingRepo.Update(outgoing);
        }

        await _unitOfWork.SaveChangesAsync(ct);
        
        // 4. Commit atomic updates dynamically
        await ((dynamic)transaction).CommitAsync(ct);
        return Ok(ApiResponse<object>.Ok(null, "IMEI successfully assigned."));
    }
    catch (Exception ex)
    {
        await ((dynamic)transaction).RollbackAsync(ct);
        _logger.LogError(ex, "Failed to bind IMEI.");
        return StatusCode(500, ApiResponse<object>.Fail(ex.Message));
    }
}
