01. Business Context & System Disruption Audit

VVS COMPUTERS (16225357 Canada Inc.) is an electronics reseller specializing in high-demand refurbished and used smartphones, tablets, and mobile accessories. Our administrative team manages wholesale purchasing and inventory distribution across various marketplace endpoints, including Best Buy Canada and Amazon.

The core technical friction within our existing system originates from an architectural flaw introduced by a third-party contractor. The contractor-built system attempted to automate IMEI (International Mobile Equipment Identity) number allocation on webhook arrival prior to physical picking.

This algorithmic failure created substantial "ghost inventory", causing double-allocations, marketplace penalties, and critical threats to our Best Buy and Amazon account health.

To resolve this, we are executing a complete system migration. The new Sovereign IMS forces a strict Human-In-The-Loop inventory verification stage, requiring a physical scan and manual allocation during processing to ensure total R2-compliance.

 * Core State Verification and Manual IMEI Allocation Engine

function verifyAndAllocateIMEI(orderId, sku, scannedIMEI) {
  var ss = SpreadsheetApp.getActiveSpreadsheet();
  var inventorySheet = ss.getSheetByName("Inventory");
  var orderSheet = ss.getSheetByName("Orders");

  var inventoryData = inventorySheet.getDataRange().getValues();
  var orderData = orderSheet.getDataRange().getValues();

  var imeiFound = false;
  var rowToUpdate = -1;

  // 1. Perform exhaustive scan of local inventory dataset for physical existence
  for (var i = 1; i < inventoryData.length; i++) {
    var imeiInRow = inventoryData[i][0].toString().trim();
    var skuInRow = inventoryData[i][1].toString().trim();
    var statusInRow = inventoryData[i][2].toString().trim();

    // Status indices: [AVAILABLE, SOLD, RESERVED]
    if (imeiInRow === scannedIMEI && skuInRow === sku) {
      if (statusInRow === "AVAILABLE") {
        imeiFound = true;
        rowToUpdate = i + 1; // Correct index tracking offset
        break;
      } else {
        Logger.log("WARNING: Physical IMEI detected but status is currently marked as: " + statusInRow);
        return "ERROR_STATUS_CONFLICT";
      }
    }
  }

  if (!imeiFound) {
    Logger.log("ERROR: Scanned IMEI " + scannedIMEI + " does not exist in localized master database schema.");
    return "ERROR_NOT_FOUND";
  }

  // 2. Perform atomic inventory status transition to ensure single-binding integrity
  inventorySheet.getRange(rowToUpdate, 3).setValue("SOLD");
  inventorySheet.getRange(rowToUpdate, 4).setValue(new Date()); // Immutable timestamp injection
  inventorySheet.getRange(rowToUpdate, 5).setValue(orderId);

  // 3. Update master order line item status for R2 Compliance verification chain
  for (var j = 1; j < orderData.length; j++) {
    if (orderData[j][0].toString().trim() === orderId && orderData[j][1].toString().trim() === sku) {
      orderSheet.getRange(j + 1, 3).setValue("ALLOCATED");
      orderSheet.getRange(j + 1, 4).setValue(scannedIMEI);
      break;
    }
  }

  return "SUCCESS_ALLOCATION_COMPLETE";
}

06. Dynamic Sovereign GUI-Based Key Rotation Engine

Enterprise Enabled

To address our customer's request for absolute autonomy without requiring developer intervention, we have designed a **Dynamic Key Rotation Engine** integrated directly into our secure administrative dashboard. This is fully possible and follows zero-trust compliance standards!

The architecture utilizes an encrypted settings provider backed by MySQL, loaded at runtime via standard .NET Dependency Injection and exposed securely through Angular 19.

🛡️ Zero-Trust Dynamic Reload Architecture

  • • Database-Backed Cryptography: All keys (JWT Security Key, SP-API tokens, Shopify tokens) are stored in our SystemSettings table, encrypted using AES-256 with an environment-level seed.
  • • Dynamic Reloading: ASP.NET Core dynamically checks settings state changes. Key updates do *not* require restarting the web server or IIS instances.
  • • Pre-flight Connection Verification: When a user updates API keys in the GUI, the backend runs an asynchronous "pre-flight" test to confirm valid connections before committing the rotation.

⚙️ Dynamic Key Rotation API Gateway Reference 

/// <summary>
/// Sovereign Key Rotation API Endpoints
/// Enforces Multi-Factor Authorization & Live Connection Pings
/// </summary>
[Authorize(Roles = "Administrator")]
[ApiController]
[Route("api/admin/keys")]
public class KeyRotationController : ControllerBase
{
    private readonly ISecureSettingsService _settingsService;
    private readonly IShopifyClient _shopifyTestClient;

    [HttpPost("rotate-shopify")]
    public async Task<IActionResult> RotateShopifyKey([FromBody] ShopifyRotationDto dto)
    {
        // 1. Verify connection to Shopify Endpoint using the new key first
        bool isValid = await _shopifyTestClient.PingWithKeyAsync(dto.NewToken, dto.ShopUrl);
        if (!isValid) return BadRequest("Invalid Shopify access token. Connection failed.");

        // 2. Commit updated token to secure database vault
        await _settingsService.RotateKeyAsync("ShopifyToken", dto.NewToken);
        return Ok(new { status = "SUCCESS", rotatedAt = DateTime.UtcNow });
    }
}
